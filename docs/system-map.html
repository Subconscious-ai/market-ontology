1Repository architecture

market-ontology is the canonical Pydantic schema for the Subconscious knowledge graph. It is the stable contract every other repo depends on.

  • Source of truth — poc_v1/ontology/schema.py: 14 node models, 18 edge models, SCHEMA_VERSION 1.6.0. The only definition of graph shape.
  • Public contract — four modules under poc_v1.ontology that consumers import directly: schema (the models), graphiti_views (ENTITY_TYPES/EDGE_TYPES/EDGE_TYPE_MAP, derived from schema), identity (company resolver, email/domain→slug), iri (canonical RDF IRIs).
  • Causal peer — causal_dag_v1/ holds causal hypotheses (CAUSES, MEDIATES, MODERATES) with NetworkX acyclicity checks. Kept separate: poc_v1 is context, causal_dag_v1 is result.
  • Generated artifacts — scripts/generate_*.py read the schema and write the *.json/*.ttl files under poc_v1/ontology/. Never hand-edit them.
  • Validation — scripts/validate_*.py + scripts/agent/ check fixtures and generated artifacts; ci.yml and symphony-gate.yml gate main.
  • Consumers — spice-harvester, ai-chatbot-native-sizzle, twenty CRM, and the burn-substrate Graphiti sidecar all pip install the package and import the public modules.

2How data flows through the system

The ontology is context for experiments, not the experiment itself. Data moves in one direction with a single loop back. The unit that moves is the (claim, source) pair — every fact carries its Evidence.

  1. Sources — a research agent, executive interviews, and web search (Exa / Tavily) produce raw material.
  2. Ingest — spice-harvester extracts node and edge candidates, each with an Evidence record, into JSONL.
  3. Validate — Pydantic validates every record against the market-ontology schema at the write boundary. Invalid records are rejected, never written. This is the only place graph shape is enforced.
  4. Graph — validated records become kg_seed/*.jsonl and a typed_graph RDF projection (IRIs minted by iri.py).
  5. Consume — UI and CRM consumers read the graph directly.
  6. Project to experiments — the ontology projects experiment inputs (Transition, StakeholderArchetype, Trait/TraitLevel, Attribute/AttributeLevel) into SuperEgo / W&B via super_ego_projection.
  7. Results loop back — the DCE engine returns AMCE / WTP results. These become normalized Estimate nodes, linked to the ontology IDs they were computed on via ExperimentRun lineage. Estimates point at ontology nodes; they never mutate them.

3How to change the schema safely

A schema change is a contract change — every consumer depends on it. Follow these steps in order. The procedure exists because parallel type systems have been introduced and reverted twice.

  1. Open an issue — state intent, acceptance criteria, and which consumers are affected.
  2. Edit schema.py — change NODE_MODELS / EDGE_MODELS and bump SCHEMA_VERSION.
  3. Regenerate artifacts — run scripts/generate_*.py. Never hand-edit the generated *.json / *.ttl files; they must match the schema.
  4. Add a fixture — add or update a kg_seed fixture and register it in scripts/validate_kg_seed.py::FIXTURES.
  5. Breaking change? If the change breaks existing consumers, coordinate them: open paired PRs in spice-harvester and ai-chatbot-native-sizzle in the same round, cross-linked. If compatible, skip to step 7.
  6. Coordinate consumers — land or stage the consumer PRs so nothing reads a shape that no longer exists.
  7. Run scripts/agent/validate-fast.sh — validators plus the full unit suite. Fix everything before pushing. Do not use --no-verify.
  8. CI gate → merge — ci.yml and symphony-gate.yml must be green. Then merge.
